Functional Testing

Assures that the developed software meets the requirements.

They should not be our unique tests.

Ideal testing pyramid

The base of the pyramid are the unit tests

  • Simple
  • Fast
  • Robust

Another vision

The inverted pyramid

The base of the pyramid is unstable.

Most of the tests are manual tests

  • Complex
  • Slow
  • Fragile

Maintain a lot of end to end tests is hard

  • Long-running
  • Many resources
  • Paths hard to test
  • Uncertainity
  • Coupled
  • Not suitable for refactoring
They may take a lot of time to execute They may need a lot of resources There are patha hard to test When they fail, we don't know exactly where o why they fail. Or not inmediatly. They are so coupled with the environment They don't give the same sense of security when refactoring.

Automation

Test repetition is necessary to eliminate regressions.

With automation eliminate the waste associated with repetition.

It's difficult


Requires discipline

  • It's very tempting not to write tests.
  • It's very tempting to see tests as waste.



Requires surveillance

  • Somebody has to observe the red results
  • Someboyd has to repair the broken windows

When it's good to automate UI testing?

Automated regression testing

Automate successfull manual tests.

Take a manual recording and add assertions.

WARNING!!

Don't take as a substitute for other tests.

Don't invert the pyramid.

Automated bug regression testing

When resolving a bug, create an automated test that can be run in the buld to ensure no reappears.

Usefull in brownfield projects.

GUI testing

Automated smoke testing

A smoke test checks that some key interacions with the system work after a deployment.

If you automate them this tests are faster, comfortable and less prone to errors.

Refactoring

If you have to refactor you could cover the code with UI tests.

Usefull in brownfield projects.

Don't use as substitute for unit testing!!!

When it's NOT good to automate UI testing?

To replace other type of tests that are not made because of laziness or because they are complicated.

It's better to think how to unit test.

The generic way

casper.on("resource.received", function(request) {

        this.echo('Resource Not Found:' + request.url);

But only applicable to "direct" requests

Finding errors

casper.on("page.error", function (msg, trace) {
    this.echo("Error:    " + msg, "ERROR");
    this.echo("file:     " + trace[0].file, "WARNING");
    this.echo("line:     " + trace[0].line, "WARNING");
    this.echo("function: " + trace[0]["function"], "WARNING");
});

waitfor

Waits until a function returns true to process any next step.

casper.waitFor(1000, function check() {
    // return something true
});

waitForSelector

Waits until the selector exists in the remote DOM.

casper.waitForSelector('a_selctor', function() {
    // do something
});

Waits until the selector exists in the remote DOM.

waitUntilVisible

Waits until the selector is visible in the remote DOM.

casper.waitUntilVisible('a_selctor', function() {
    // do something
});

	this.echo("Wellcome message visible");
});

Evaluate

Evaluates an expression in the current page DOM context.

Think of the evaluate() method as a gate between the CasperJS environment and the one of the page you have opened; everytime you pass a closure to evaluate(), you're entering the page and execute code as if you were using the browser console.

example

casper.start('http://casperjs.azurewebsites.net/');

casper.then(function()
{
	this.echo(this.evaluate(function()
	{
		return document.title;
	}));
});

exercise

change the body background color of the page and then, make a capture

Solution

casper.start('http://casperjs.azurewebsites.net/');

casper.then(function()
{
	this.evaluate(function()
	{
		$("body").css("background-color", "red");
		return;
	});

	this.capture('redBody.png');
});
